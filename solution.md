### 1. 设计思路（一句话总览 + 主轴）

**一句话**：这个 Copilot 的目的是**尽可能完整地分析一条线索里的所有信息，用漏斗式分层模型给出决策建议**，从而提高销售的线索处理成功率、沉淀可复用的方法论、并减少销售在重复判断上的注意力消耗和耗时。

**主轴流程**——我把整个系统看成一条以"线索"为载体、以"反馈数据"为闭环的流水线：

```
业务背景信息（产品目录 / SOP / 禁语 / 自定义方法论）
        ↓
      一条线索进入系统
        ↓
AI Agent 做初步信息整合（Reasoning Summary）
        ↓
按需调用 Tool 查询企业内部资料（Act → Observe）
        ↓
输出分层后的关键信息（A/B/C/D 等级 + 痛点 + 缺失信息 + 风险 + 跟进话术）
        ↓
人工基于分层后的信息做下一步处理
        ↓
处理结果（成交 / 未成交 / 待跟进）通过 openid 回到数据库
        ↓
Trace + openid + 反馈结果沉淀为可复盘的迭代数据 → 迭代方法论 / Skill / Tool
```

**核心抽象按职责拆为四块**，每块只做一件事：

- **Skill** 用以封装"业务 prompt"。它让业务人员可以按照自己的方法论持续迭代分析逻辑，而不必动后端代码。
- **Tool** 让模型能拿到企业内部数据库的内容（当前只接了 `query_product`，对应本地产品目录）；后续迭代时它也是接入外部能力的入口（如 CRM、合规库、行业知识库等）。
- **Runner** 是执行器，把上面三者串起来跑出 ReAct 循环，让业务人员在前端能直观看到结果。
- **Trace** 记录全过程，**同时服务三种用途**：debug、质检、迭代。它是把"AI 的判断"转变成"可审计、可累积、可改进的资产"的关键。

**为什么这样拆**：我不想让 LLM 直接"端到端"输出答案。LLM 容易在没有约束时编造内容（价格、案例、能力），所以我把每一类知识 / 能力 / 决策结构都显式地放到一个抽象里：业务约束放 Skill / 资产文件，外部访问能力放 Tool，循环控制放 Runner，可审计性放 Trace。这种拆法的副作用是组件多了几个，但带来的好处是**任何一块出问题都能定位到独立模块**，业务人员要改提示词不用碰代码，工程师要加新能力不用碰 prompt。

设计原则上有几条贯穿始终（详见 `design_points.md`）：单一职责、所有事实必须带 `source_id`、Trace 只暴露可审计摘要不暴露完整 CoT、Runner 用兜底分支保证 API 契约稳定、Skill 是权限边界、未授权或失败工具自动进入 `tool_iteration_requests` 反馈给供应商团队。

### 2. 为什么这样定义 Skill `lead-scoring-followup`

**边界划分的理由**：Skill 在我的设计里同时承担两个身份。第一个身份是**技术上的权限边界**——`allowed_tools` 是这条 Skill 可以调用的工具白名单，Runner 在 Act 前做白名单检查，越界请求会被记账到 `tool_iteration_requests` 并强制 `needs_human_review=True`。第二个身份是**业务上的方法论版本控制工具**——SKILL.md 的 frontmatter 带 `version`，正文是业务人员定义的分析方法论；改了之后 bump 版本号，Trace 里 `prompt_version` 会记录本次执行用的是哪一版。这样一条线索分析出问题时，能精确回滚到具体哪一版方法论。

我希望 Skill 这层最终是**业务人员自己写、自己迭代**的。所以在扩展里我加了 `data/custom_playbooks/`：销售可以把自己摸索出的高转化话术、踩坑总结写成 markdown 沉淀进来；Runner 在构造 system prompt 时会把这些 playbook 一并加载。这本质上是让 Skill 成为**业务方法论的可共享、可版本化的承载体**：一线销售把方法论写成文档 → 通过 playbook 自动作用于 AI 的判断逻辑 → 团队里其他销售直接受益于这份方法论。

**当前 SKILL.md 的 `instructions` 里写了什么**（按重要性）：

1. 角色定位与服务对象（线索运营助理 / 服务销售经理）—— 决定输出风格
2. 每一轮的输出契约（必须返回 JSON，含 `reasoning_summary` 和 `route_decision`）
3. 何时调 `query_product`、何时直接进入 Answer 的判定规则
4. **防编造硬约束**：每条事实必须带 `source_id`，否则改写为"需人工确认"
5. 最终 `answer` 的字段约定（lead_tier / pain_points / draft_reply / next_actions / triggered_rules / evidence …）

第 4 条是评审最关心的，我把它放在 SKILL.md 比较显眼的位置，因为这是模型最容易"善意越界"的地方。

**未来加 Skill 是否支持**：Loader 设计上已经支持。`load_all_skills(skills_root)` 会扫描 `skills/` 目录下每个含 `SKILL.md` 的子目录，把它们解析成 `Skill` 对象按 name 索引；并且会检测重名报错。所以未来要加 `objection_handling`、`customer_onboarding` 等 Skill，只需要 `skills/objection-handling/SKILL.md` 这样新建一个子目录、写好 frontmatter 和正文即可，Loader、Runner、Tool 注册表完全不需要改代码。如果不同 Skill 需要不同 Tool 组合，也只是 frontmatter 里 `allowed_tools` 的差异——这种"配置即权限"的设计意味着同一个 Tool 池可以被多个 Skill 选择性复用。

### 3. 为什么需要 `query_product` 这个 Tool

**两个核心诉求：防幻觉 + 优秀的可插拔可维护能力。**

**不用 Tool 让 LLM 直接靠训练知识回答会有两类问题。** 一类是幻觉——LLM 没见过我们的产品目录，价格、能力、案例都可能编造，提高幻觉率。另一类是不可迭代——企业的产品 SKU、定价策略、客户案例每周都在变，如果把这些知识写进 system prompt 或全文塞进上下文，每改一次都要工程介入改 prompt，并且很快会撑爆 context 导致 context rot。Tool 化之后，知识库变更不需要重训也不需要改 prompt，只要数据库内容更新即可，Tool 调用拿到的就是最新版本。这一层"freshness"对销售场景特别重要——产品和案例库是高频变动资产。

**为什么不用向量库 + 直接 RAG 替代 Tool**——这是评审可能问的反驳点。向量库适合"找相似文本"，但销售场景下检索往往是结构化属性匹配（行业 + 公司规模 + 阶段 + 异议类型），SQL/Tool 调用比向量相似度精准很多，并且 Tool 的输入参数本身就是 LLM 的结构化推理结果，可解释性更强。向量库可以作为 Tool 内部的实现细节，但不该替代 Tool 这一抽象层。从迭代成本看也一样：直接全塞或纯向量库的方案，到后期的迭代成本都会比 Tool 化高得多。

**为什么必须让 Tool 返回带 `source_id` 的结构化结果**——三个原因：

1. **防编造闭环**：Skill 里硬约束"事实必须带 source_id 否则改写为'需人工确认'"。这条约束只有在 Tool 返回结果带 source_id 时才能落地，没有 source_id 模型就有自由发挥的空间。
2. **可追溯性（销售 + 审计两类受众）**：销售看到 source_id 能跳转到原始成交案例做参考话术；审计能追溯每一次 AI 决策依赖的具体证据，验证 Copilot 是否有效、各部门是否按 SOP 执行。（管理层关心的是聚合后的指标趋势——见 Q9——而不是单条 source_id 本身，所以 source_id 的直接受众是两类。）
3. **反馈闭环**：Trace 里记录了"这次回答用了哪些 source_id"，再叠加 openid 回写的成交结果，就能反向算出"哪些案例真正促成了转化"，给检索打分、做主动学习。如果不返回 source_id，这条数据链直接断掉。

### 4. ReAct 各阶段如何体现

**为什么用 prompt 化的 JSON 字段而不是模型原生 function calling**——三点：成本低（不依赖特定厂商的 FC API）、易于构建（任何 LLM 都能跑）、足以满足业务需求。代价是 JSON 解析可能失败——我们在 prompt 里用 few-shot + JSON mode，并在解析失败时降级为"纯文本输出 + 强制 `needs_human_review=True`"，避免脏数据流出。原生 FC 的 schema 保证更强，未来如果稳定性出问题可以无缝切换，但当前 demo 阶段这套足够。

**Act**：[TODO 候选人补充：什么场景才调 `query_product`？传什么参数？]

**Observe**：Tool 返回的结构化结果**完整塞回上下文，而不做摘要**。原因是返回字段本身已经很紧凑（3 条 × ~200 字，预算可控），完整塞回去能最大化降低幻觉率，且关键的 `source_id` 字段不会因摘要而丢失。如果未来 Tool 返回数据量上涨（比如要返回 20 条案例），再考虑加摘要层。多语言场景下不同 tokenizer 的 token/字符比差异较大，未来要支持双语客户时需要重新评估这个上限。

**Answer 阶段的字段划分围绕三类不同受众**——销售看 `draft_reply`，管理看 `next_actions` 和 `lead_tier`，审计看 `evidence` 和 `source_id`。每类受众读的东西完全不同，所以字段切分按受众而不是按"信息类型"。

- **`draft_reply`（≤200 字）**：销售不需要理解 AI 推理，拿来改几个词就能发出去。命名上特意叫"草稿"而不是 `suggested_reply`，因为"草稿"暗示销售仍然是主体、AI 只是辅助。200 字上限基于历史 IM/邮件外发消息的 P50 长度做的约束（[占位：补具体数据来源]）。
- **`next_actions`**：必须是具体可执行的动作列表（建议带 `priority` 和 `due_in_hours` 字段以贯彻"最小化思考路径"），不是抽象方向。这么做是为了最小化初级销售的思考成本——人的注意力有限时，让销售直接做 AI 做不了的执行动作（打电话、上门、走审批），最大化系统运转效率，避免他还要在"先做哪个"上多想一步。

### 5. 哪些上下文会传给模型

**切分原则**：system prompt 放"不变的全局规则与能力"，user message 放"变的、单条线索的事实与历史观察"。这条边界一旦清晰，每条信息往哪里塞就不再需要逐条权衡。

**system prompt 里塞什么**：

- **Tool 描述**：每个被 Skill 白名单允许的 Tool 的 name / description / params schema / 返回结构示例。
- **SOP**：团队的销售标准动作（什么阶段该问什么问题、什么客户分层下推荐什么产品）。
- **约束**：Skill instructions、forbidden_claims、字段格式契约等硬约束。

**user message 里塞什么**：

- **初始 user message**：原始线索内容（客户提问、可观察的客户画像字段等单条线索事实）。
- **每一轮工具调用结果**：按 Act → Observe 顺序累积塞回，完整塞回不做摘要（论证见 Q4）。

**哪些故意不塞，为什么**：

- **`openid` 等内部 DB 主键 / 内部标识符**：模型不需要读取这些标识符就能完成推理，塞进去只会占 token 且增加 PII 泄露风险。它们的归属是"内部流程层"——只在 Trace 写入和数据库回写时用到，本身不是模型推理的输入。

注意 `source_id` **不在**"故意不塞"清单里——它会随 Tool 返回结果进入 Observe，让模型在 answer 里能引用（这是 Q3 防编造硬约束能落地的前提）；同时它也写入 Trace 供审计追溯。"模型可见" 和 "审计可追溯" 是两个独立但兼容的用途。

### 6. 如何避免 AI 编造价格 / 周期 / 案例 / 产品能力

按"事前 / 事中 / 事后"三层纵深防御组织——单层都可能被绕过，叠在一起才能压低编造率。

**Prompt 层（事前约束）**：约束逻辑详见 Q3——事实必须带 `source_id`，否则改写为"需人工确认"。这是写在 SKILL.md instructions 高优先级位置的硬约束。

**Trace 层（事后兜底 + 迭代依据）**：审计能根据 Trace log 还原模型的完整运行逻辑——每轮的 `reasoning_summary`、调用了哪些 Tool、引用了哪些 `source_id`、最终输出了哪些字段。这一层有两个用途：

- **事后兜底**：发现违规时能定位到具体轮次、`prompt_version`、Tool 返回数据，知道是哪一步出了问题。
- **迭代依据**：把高频违规模式（比如"模型经常在没有 case 数据时编造客户案例"）回流到 SKILL.md 或 `forbidden_claims.md` 的更新里。

**`forbidden_claims.md`（禁语清单）**：列举常见违规行为类目，可根据行业现实的法务和业务禁令持续维护。该文件初稿由 AI 辅助生成（详见 Q10 的 AI 使用方式声明），后续由业务/法务校对补充。它作为独立文件存在而不是合并进 SKILL.md，是因为禁语清单的更新节奏（法规、行业规范变化）和销售方法论的迭代节奏不同步，解耦后两边可以独立维护。

### 7. ReAct 上限为什么是 3 轮

**3 是经验值——MVP 阶段先用这个数走通流程，不是数据回归出来的。**

这是一个**类 deep research 的架构**：理论上 ReAct 轮数越多回答质量越高，因为模型可以反复调 Tool 补信息。但**回答质量的天花板被数据库完整度卡住**——一旦数据库覆盖不到某条线索的关键事实，多调几轮 Tool 只是空转，边际收益快速衰减。所以"3 轮"的本质是在 **"质量上限 vs 成本/延迟"** 之间取一个 MVP 阶段够用的折中：

- **设得太少（比如 1 轮）**：模型只能 query 一次就被迫 answer，无法做 follow-up 检索（先 query 产品后再 query 案例的链路就断了），质量打折。
- **设得太多（比如 10 轮）**：因为质量已被数据库完整度封顶，额外轮数带来的边际收益小，但成本、延迟、注意力漂移线性上涨。

**触发上限时 Runner 怎么收敛**：

- **仍旧强制出 answer**，保证 Copilot 的可用性——不允许出现"跑了几轮没结论"的用户体验。
- **第 3 轮如果模型还想 call_tool，Runner 不允许，直接走 answer 分支**。当前 MVP 不做"再 reasoning 一次"这种中间补救态。
- **未来迭代方向**：等真实流量数据上来后，可以做动态上限（按线索复杂度判定）、或者把复杂线索拆给多个 Skill 串联处理。这些都属于 MVP 阶段明确的技术债。

### 8. Trace 设计的取舍

**为什么不展示完整 CoT**：两个原因叠加。一是**题目本身的要求**；二是**符合业务逻辑——Trace 的三类受众（销售 / 管理 / 审计）都只需要看"重要消息"**，完整 CoT 是模型的"自言自语"，对任何一方做决策都没有增量价值，反而会稀释他们对关键字段（`source_id`、`triggered_rules`、`needs_human_review` 等）的注意力。Q4 已经讲过字段切分是按受众切的，Trace 这一层延续同一个原则：只暴露各类受众真正需要的信息。

**保留了哪些字段？舍弃了哪些**（基于 `backend/schemas.py: TraceStep` 与 `backend/runner.py` 实际实现）：

保留的字段——

- **`step_id`**：步骤序号，决定时间线
- **`step_type`**：`init` / `reasoning` / `act` / `observe` / `answer` / `max_turns_guard` / `error` / `warn`——分类清晰，便于过滤
- **`skill_name` + `prompt_version`**（在 `init` 步骤）：审计能精准定位"这条 answer 是哪一版方法论给出的"
- **`context_summary`**：每步的上下文摘要（如 "system prompt N chars · skill=... · max_turns=3"），不是完整 prompt
- **`tool_name` + `tool_input` + `tool_output`**（在 `act` 步骤）：工具调用的完整链路，用来审计模型依赖的具体数据
- **`output_summary`**：**决策摘要，非完整 CoT**——这是 Trace 设计最关键的取舍点：`reasoning` 步骤的 `output_summary` 是模型的简短结论而非整段推理；`answer` 步骤的 `output_summary` 只抓 `lead_tier` / `needs_human_review` / `triggered_rules` 这几个最重要的判定字段
- **`latency_ms`**：单步延迟，用于性能审计与回归
- **`error`**：各类错误（LLM 调用失败、JSON 解析失败、Tool 越权等）独立记录

舍弃的字段——

- **完整 CoT**：`reasoning` 步骤里只存 `output_summary`，模型的完整自言自语不写盘——这是"不展示完整 CoT"在数据层的落地
- **`openid`**：openid 只作为 `AnalyzeResponse` 顶层字段 echo 回前端做 join key，不进入 TraceStep（与 Q5"内部 ID 不给模型"的口径对齐——同理 openid 也没必要存进每一步 Trace）
- **`lead_text` 原文的二次拷贝**：原文只通过 user message 进入模型，不在 Trace 里再存一份，避免冗余存储和 PII 二次落地

特殊设计点——`max_turns_guard` 作为独立 `step_type`，明确标记"因达到 ReAct 上限而强制走 answer 分支"。审计能立刻识别这种被截断的样本，与 Q7 描述的收敛策略闭环对齐。

**审计可复现性（弱复现口径）**：评审人能从 Trace 定位到一条 answer 是由 **"哪一版 prompt（`prompt_version`） + 哪些 Tool 调用（`tool_name` / `tool_input` / `tool_output`） + 哪些决策摘要（`output_summary`） + 是否被强制截断（`max_turns_guard`）"** 得出的，足够判断模型的行为是否合规、是否触犯禁语、是否引用了正确的 source_id。

不追求"给同样输入能 replay 出完全相同 answer"的强复现——LLM 本身有温度，强复现在工程上代价过大且对审计无增量价值（审计关心的是"决策路径是否合规"而不是"输出字符是否逐字一致"）。

### 9. 如果给真实业务使用，你会用哪些指标判断它是否有效

**两个主指标——必须组合看，不能单独评估**：

1. **人工接受率**：代表 Copilot 的有效性 + 被销售的实际接受程度。这是产品"被用没用"的直接信号。
2. **跟进转化率**：代表 Copilot 是否切实提高了工作效率和回应质量。这是产品"真带来业务增量"的信号。

**反作弊设计——为什么必须两个一起看**：

如果只盯人工接受率，会有"假阳性"——销售可能嫌改稿麻烦、直接复制 AI 草稿发出去，让接受率虚高。所以要叠加交叉信号识别这种偷懒模式：

> **如果人工接受率高 + 跟进转化率下降 + 审核未通过率上升** → 说明是销售在用 AI 草稿走过场，并不是 Copilot 真的好。

这个组合模式一旦出现，必须回看具体是 prompt 退化、Tool 数据陈旧、Skill 方法论失效，还是销售操作姿势的问题——不能因为人工接受率高就停止迭代。Q8 的 Trace 字段（`prompt_version` / `tool_input` / `tool_output` / `output_summary`）正是用来支持这种归因分析的。

**为什么 MVP 阶段不引入更多指标**：候选清单里的客户回复率、人工复核工时下降、误判率、编造内容率都有价值，但 MVP 阶段先收敛到能直接判断产品价值的最少指标——指标过多会稀释团队注意力、也会带来"指标好看但产品没用"的伪迭代风险。等 2 周后接入真实流量、Q12 的 dashboard 上线，再分批引入辅助指标做能力分级。

### 10. AI 使用方式声明

**AI 辅助的部分**：

- **大部分代码**由 AI 协助生成（具体边界可结合 commit log 与 `design_points.md` 一同评审）
- **`forbidden_claims.md`** 初稿由 AI 生成（见 Q6 说明），后续由业务/法务校对补充
- **`solution.md` 的所有题目（Q1–Q12 + 风险与限制）**——由我本人提供回答要点、设计判断、价值取舍，再经由 AI 做结构化语言改写（拆分小节、补全衔接、统一措辞、对齐跨题引用）。每节的扩展点都在对话中逐条与我确认；任何 AI 单方面延伸超出我口述范围的内容，我都在审阅阶段做了删改或留 `[TODO]` 待补。

**本人写/设计的部分**：

- **架构与技术栈选择**：Skill / Tool / Runner / Trace 的四层职责拆分；FastAPI + 原生 JSON 字段而非模型 function calling 的选型
- **ReAct tool call 层数**：`max_turns=3` 的判定（理由见 Q7）
- **`openid` 字段的设计**：作为对外业务 ID，作为 `analysis_log` 与 `feedback_log` 的 join key 支撑反馈闭环（与 Q3 的 source_id 闭环结合形成"事实溯源 + 成交结果回流"的完整数据链）
- **Skill 系统的设计目标**：业务方法论的可版本化、可共享承载体，同时承担权限边界（`allowed_tools` 白名单）

**关键决策——AI 建议被我否决/修改的例子**：

- **Trace 字段精简**：AI 最初提议了一份偏冗余的 Trace 字段清单，我修改成目前的精简模式（见 Q8 字段列表）。否决/简化的理由就是 Q8 描述的取舍——完整 CoT 不写盘、内部 ID（openid）不进 TraceStep、每步只留 `output_summary` 作为决策摘要——以匹配"审计快速读判断"的目标而不是"全量记日志"。

### 11. 可选扩展（如果你做了）

实际做了两个扩展——`POST /api/feedback` 反馈接口 和 `PUT /api/playbooks/{name}` 自助 playbook 上传接口。两者背后是同一条信念：**一款产品不是一出世就完美的，只有靠整体的业务数据和用户反馈反复打磨优化才能产生好产品**。这两个接口就是把"打磨"这条路径在产品层显式落地。

**扩展一：`POST /api/feedback`（销售回写成交结果）**

- **解决的问题**：Copilot 的分层 / 跟进建议是否真的有效？没有反馈闭环就无法度量——所谓"AI 提升销售效率"只是凭感觉。
- **服务的业务目标**：让产品的有效性可度量、可迭代。Feedback 接口是反馈闭环的入口，下游直接支撑 Q9 的两个主指标（人工接受率 + 跟进转化率）的计算，并与 Q3 的 `source_id` 串成"事实溯源 + 成交结果回流"的完整数据链。
- **优先做它的理由**：所有其他指标和迭代都依赖反馈数据。没有 feedback，Q9 的转化率根本算不出来，Q3 的"反馈闭环"也没数据源——它是其他扩展的前置条件。

**扩展二：`PUT /api/playbooks/{name}`（业务人员自助维护 playbook）**

- **解决的问题**：业务方法论必须由最贴近业务的人来定义，但他们既不写代码也不动 prompt。如果方法论迭代必须走工程团队的发版节奏，方法论永远跟不上业务变化的速度。
- **服务的业务目标**：让**一线业务人员**自行迭代筛选线索的方法论——他们是离产品和使用场景最近的人，能给出最好的修改建议。**允许个性化配置是未来 AI 产品好用的一大关键。**
- **优先做它的理由**：MVP 阶段如果不预留 playbook 接入位，后期再拆"硬编码在 SKILL.md 里的方法论"会非常贵。提前打通自助通道，相当于给"业务持续自迭代"留了一条独立于工程团队的路径。

**两者的关系**：feedback 提供"实际效果数据"，playbook 提供"业务人员根据效果数据自助优化方法论的能力"——前者是观测，后者是调控。两个扩展合起来才构成完整的"测量—修正"闭环。

### 12. 后续迭代计划（72 小时 / 2 周 / 1 个月分别会做什么）

下面列的都是**之前已经规划但 MVP 阶段没做的功能**。分层标准是"对真实业务数据的依赖度"——72 小时不依赖真实数据、靠工程自查就能完成；2 周依赖少量真实流量做假设验证；1 个月依赖足够量的反馈数据做平台化判断。

**72 小时（紧迫 / MVP 兜底——不依赖真实数据）**：

- 回填 Q4 的两个 TODO（Reasoning Summary / Route Decision 字段明确化、Act 触发场景的显式判定规则）
- Tool 错误兜底强化：当前 Tool 失败统一走 `needs_human_review`，应区分"可重试错误"和"永久失败"，避免间歇性网络抖动也强行人工复核
- Trace 持久化升级：当前 `analysis_log.jsonl` 只是 demo 级别，需要落到结构化存储（SQLite 起步），支持按 `prompt_version` / `skill_name` 切片回放
- Prompt 回归评估集：手动构造 20–30 条典型线索作为黄金集，每次改 SKILL.md / `forbidden_claims.md` 时跑一遍，防止改动悄悄退化

**2 周（业务上量需要——需要真实流量进来才能验证方向）**：

- 接 CRM Tool：让 Copilot 能查客户的历史互动 / 现有合同 / 关键联系人，而不只是产品目录
- 加 `case_library` Tool：让 `source_id` 引用扩展到真实成交案例库（不只是 `PROD-xxx`，还能引 `CASE-xxx`）
- 新增 Skill：`objection_handling`（异议处理）、`customer_onboarding`（成交后跟进）——这一步顺带验证 Skill Loader 在多 Skill 场景下的稳定性
- A/B 框架：不同 `prompt_version` 同时跑，对比人工接受率和跟进转化率（Q9 的两个主指标）

**1 个月（平台化——依赖大量真实数据做能力分级）**：

- 多角色权限：销售 / 主管 / 审计看到的 Trace 字段不同（同一份 Trace，按角色差异化呈现）
- Skill 市场：业务团队自助上线 / 订阅 / 收藏 Skill；不同部门复用对方的方法论
- Playbook 推荐：根据 feedback 数据反向算"哪些 playbook 真的促成了转化"，主动推荐给销售
- 合规审计后台：Trace 可视化 + 触犯禁语自动告警 + `triggered_rules` 聚合统计
- 跨 Skill 协作：一条线索经过多个 Skill 接力（初步分层 → 异议处理 → 成交后跟进）

**贯穿三个阶段的核心节奏**：所有迭代决策都依赖**真实业务数据反馈**来定优先级——MVP 上线后才知道哪些痛点是真实存在的、哪些是被想象出来的伪需求。72 小时做的是必须先补的工程债；2 周开始用真实流量验证"假设性扩展"的价值；1 个月才能在已经积累的反馈数据上做平台化判断，避免在没有数据支撑的情况下过早过度设计。

## 风险与限制

**数据层简化**

全部用 JSON 文件存储（`leads.json` / `product_catalog.json` / `analysis_log.jsonl` 等），**没有使用 database**。数据量上去后无法应用在真实生产环境，且**没有做深入的数据管理**——没有数据版本、没有 schema migration、没有并发写入保障、没有备份恢复机制。

**接口层简化**

**Tool call 没有用标准的 OpenAI function calling 格式**，用的是 prompt 化的 JSON 字段（这是 Q4 里讲过的有意选型，但要诚实承认它的代价）——放弃了 schema 强约束、放弃了 FC 工具链下游的自动校验、参数辅助、错误归因等能力。JSON 解析失败的概率始终大于 schema-enforced FC。

**可观测性缺失**

**没有 dashboard**——Q9 提到的两个主指标（人工接受率、跟进转化率）目前只能直接查 `analysis_log.jsonl` 手算，没有可视化、没有告警、没有趋势看板。Trace 也没有结构化检索界面，审计只能 grep。

**评估能力缺失（最大短板）**

**没有真实业务数据作为测试集 eval 系统可用**——所有 prompt 调整、Skill 修改、Tool 增删都只能"看着像对的就发"，没有可回归的黄金集。这意味着任何迭代都有"看着好其实变差了"的潜在风险。这是 MVP 阶段最致命的局限，必须靠真实业务流量上线后回流数据来填补。

**整体定位**

当前只是一个**简单可用的 MVP**——能跑通端到端的 ReAct 流程、能产出结构化结论、能记录可审计的 Trace，但所有"能否扩展到生产"的关键能力（数据库、监控、eval、合规审核）都还是空白。

**真上线时会变什么**（与上面每条短板一一对应）：

- 数据层 → 引入关系型数据库（量大时叠加向量 DB），补齐完整的数据生命周期管理
- 接口层 → 视稳定性需求评估是否切换到原生 function calling
- 可观测性 → 上 dashboard + 指标告警 + Trace 检索后台
- 评估能力 → 用真实流量回流建立 eval 集，让 prompt 迭代由"凭感觉"切换到"数据驱动"
